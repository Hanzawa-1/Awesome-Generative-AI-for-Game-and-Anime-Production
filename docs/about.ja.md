# 概要・運用方針

## これは何か

**ゲーム・アニメ制作**に関連する生成AI / MLの **タスク** を厳選し、「領域 → タスク」のツリーで整理した
カタログです（paperswithcode系のタスクブラウザの構造に着想を得ています）。各タスクでは次を追跡します。

- **オープンソースのモデル・論文** — カタログの大半を占めます（それぞれarXiv論文および/または実在する
  コードリポジトリに紐づきます）。
- 実運用されている **商用・業界ツール**。

## 項目の発見方法

GitHub Actionsのジョブが **週に1回** 実行され、LLMリサーチエージェント（Google Gemini と OpenRouter を
切り替え可能）が読み取り専用の探索ツールを使って新着を収集します。

- **arXiv** — タスクごとの最新論文
- **Hugging Face** — 注目の論文・モデルリリース
- **GitHub** — 新規・活発なリポジトリ
- **DuckDuckGo** — 一般的なWeb検索
- **Semantic Scholar**（およびベストエフォートの `paperswithcode.co` / Civitai）— 補助

エージェントは週ごとに対象領域をローテーションし、網羅性を高めます。

## データを信頼できる理由

エージェントは項目を **提案するだけ** です。その後、決定論的なパイプラインが次を行います。

1. すべてのレコードを厳密なスキーマとタクソノミーで **検証** します。
2. 正規の id と重複排除キーを **再計算** します（LLMの主張は信用しません）。
3. 主要リンクが実際に到達可能か **検証** し、オープンソース項目には arXiv id か GitHub リポジトリを必須とします。
4. 既存カタログ全体に対して **重複排除** します。
5. 空のフィールドのみを埋める **保守的なマージ** を行い、人の編集は **決して** 上書きしません。

結果はプルリクエストとして提出され、人がレビューしてマージします。サムネイルは自動抽出されます
（arXivの図 → Open Graph画像 → プレースホルダー）。

!!! warning "自動キュレーションについて"
    要約は機械生成であり、意図的に中立的な表現にしています。誤りを含む可能性があるため、これは発見の補助
    として扱い、権威ある情報源とはみなさないでください。PRによる修正を歓迎します。

## タクソノミーの編集

領域 → タスクの構造は単一のファイル `taxonomy.yml` で管理されています。タスクの追加・改名・削除は
このファイルを編集するだけで、次回ビルド時にナビゲーションと全ページが再生成されます。
